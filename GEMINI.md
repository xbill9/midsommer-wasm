# 🇸🇪 Midsommer Madness Developer Guidelines for Gemini (Flutter Version)

Welcome! This file contains developer context, architecture notes, and guidelines for working on this project.

## 🛠️ Build and Automation Commands

Use the provided `Makefile` to run, build, test, and debug this application:

* **Compile Rust WebAssembly**: `make build-wasm` (compiles the Rust physics engine to `assets/game_physics.wasm` and generates the Base64 inline helper `assets/wasm_binary.js`)
* **Local Web Development**: `make dev` or `make run` (triggers `make build-wasm` and boots up local dev server serving the web assets)
* **Run Flutter Test Suite**: `make test` (runs unit and widget tests for the Flutter wrapper)
* **Compile Android App**: `make build-apk` (triggers `make build-wasm` and compiles a debug APK via `flutter build apk --debug`)
* **Compile iOS App**: `make build-ios` (triggers `make build-wasm` and compiles an unsigned iOS app via `flutter build ios --no-codesign`)
* **Deploy to Device/Emulator**: `make install-apk` (runs `flutter install` on connected device)
* **Monitor Native Logs**: `make logcat` (attaches to active devices and streams log outputs in real-time)
* **Clean Build Directories**: `make clean` (runs `flutter clean` and clears workspace caches)
* **Start Firebase Emulators**: `make firebase-emulators` (starts local Firestore and Hosting emulator suite for isolated testing)
* **Deploy Web App to Firebase**: `make deploy` (manual bypass deployment to the production Hosting site; pushing/merging to `master` on GitHub automatically deploys to live)
* **Deploy Preview Channel**: `make deploy-preview` (deploys temporary preview hosting channel, valid for 7 days, for PR verification)
* **Check Firebase Status**: `make firebase-status` (checks current configuration rules, active project id, etc.)
* **Deploy Database Rules**: `make deploy-rules` (deploys local Firestore security rules to Cloud)
* **Fetch Cloud Logs**: `make firebase-logs` (runs `gcloud logging read` to fetch remote Firebase/GCP logs)

## 📂 Project Architecture

* **Rust / WebAssembly Physics Engine**:
  - [game-wasm/src/lib.rs](file:///home/xbill/midsommer-wasm/game-wasm/src/lib.rs): High-performance 60Hz fixed-timestep game physics, AABB & circle collision system, entity state logic, sound event queue, and active particle system written in Rust and compiled to WebAssembly.
  - [game-wasm/Cargo.toml](file:///home/xbill/midsommer-wasm/game-wasm/Cargo.toml): Declares Rust dependency targets and optimization configurations (optimized for size `opt-level = "s"` in release profile to keep WASM payload tiny).
  - [assets/wasm_binary.js](file:///home/xbill/midsommer-wasm/assets/wasm_binary.js): Autogenerated JavaScript wrapper module storing the compiled WASM binary in `WASM_BASE64` to load inline.
* **Web Assets**: The core game frontend, UI, and media scheduler are located inside the `assets/` directory:
  - [index.html](file:///home/xbill/midsommer-wasm/assets/index.html): Entry point, mobile-first touch UI canvas, and custom high-score name entry forms.
  - [game.js](file:///home/xbill/midsommer-wasm/assets/game.js): Main rendering cycle, user keyboard/touch event managers, custom procedural Web Audio synthesizer, and coordinating interfaces that synchronize with the Rust-WASM engine.
  - [index.css](file:///home/xbill/midsommer-wasm/assets/index.css): Styling, mobile scaling guidelines, screen impact shaking rules, and Swedish retro aesthetics.
* **Wrapper Framework**: Located in the main project, wrapping the web files inside a full-screen, landscape-locked `WebViewWidget`:
  - [main.dart](file:///home/xbill/midsommer-wasm/lib/main.dart): Sets up fullscreen sticky immersive mode, configures the `WebViewController` with unrestricted JavaScript, initializes Firebase, handles Firestore leaderboard reading/writing via the `LeaderboardChannel` JavaScript channel, and falls back to local `SharedPreferences` if unconfigured or offline.
  - [pubspec.yaml](file:///home/xbill/midsommer-wasm/pubspec.yaml): Registers dependencies like `webview_flutter`, `firebase_core`, `cloud_firestore`, `shared_preferences`, and configures the `assets/` directory.
* **Firebase Configuration & CI/CD**:
  - [firebase.json](file:///home/xbill/midsommer-wasm/firebase.json) & [.firebaserc](file:///home/xbill/midsommer-wasm/.firebaserc): Configuration rules and target project definition for Firebase Web Hosting.
  - [google-services.json](file:///home/xbill/midsommer-wasm/android/app/google-services.json): Placeholders required to compile Google Services on Android, replaceable with production credentials.
  - GitHub Workflows: Automatically deploys code to Firebase Hosting. Located at [.github/workflows/](file:///home/xbill/midsommer-wasm/.github/workflows/).
    * [firebase-hosting-pull-request.yml](file:///home/xbill/midsommer-wasm/.github/workflows/firebase-hosting-pull-request.yml): Deploys testing preview channel on pull requests to `master`.
    * [firebase-hosting-merge.yml](file:///home/xbill/midsommer-wasm/.github/workflows/firebase-hosting-merge.yml): Deploys live channel on merges or pushes to `master`.

## ⚠️ Key Instructions for Gemini / Antigravity

1. **Rust / WebAssembly Modifications**: If you modify `game-wasm/src/lib.rs` or any Rust file, you **MUST** run `make build-wasm` to recompile the WASM binary and update `assets/wasm_binary.js` so the JavaScript layer can load your changes.
2. **Memory Alignment & Extern C Interfaces**: Keep in mind that WebAssembly exports must use `#[no_mangle] pub extern "C"` and only use WASM-compatible primitive types (`f32`, `i32`, etc.). Avoid complex Rust structs in the public API; instead, use indices or flat getters/setters (e.g. `get_particle_x(idx)`).
3. **WASM Caching & State Persistence**: The WebAssembly instance maintains game state in Rust static memory (like `GAME_STATE` or `PARTICLES`). When changing levels or restarting, make sure `init_level_wasm` is called to clean or reset static state properly.
4. **Asset Modifications**: If you modify `game.js`, `index.html`, or `index.css` under the `assets/` directory, the changes will be automatically picked up on the next hot restart or Flutter build.
5. **Web Audio Compatibility**: The Android WebView requires user interaction to initialize the Web Audio API context. Ensure `AudioContext` resumes on first touch/click.
6. **Responsive Canvas**: Keep the target aspect ratio in mind when tweaking layouts. Touch controls should dynamically scale.
7. **Procedural Audio Synthesis**: Do not use raw audio file formats. Audio effects and background tracks are procedurally generated in `assets/game.js` via the Web Audio API's `SoundEffectsManager`. Scales and tempos are programmatically shifted to suit the atmosphere of the active level.
8. **Level-Specific Classes**: Each level features specific enemy types subclassed from the generic `Enemy` parent class (e.g., `Shopper`, `Drunkard`, `CandyKid`, `ZappaFan`, `VolvoCar`, `DalarnaHorse`, `Elk`, `Guard`, `Raver`, `ABBAbot`). Ensure any game balance tweaks or new mechanics honor these subclass extensions.
9. **Firebase Integration & Leaderboard Bridge**: High scores are synchronized via `LeaderboardChannel` using standard JSON communication. If Firebase fails to initialize or is offline, the systems fallback to `SharedPreferences` cache (on mobile) or `localStorage` (on web) cleanly without breaking gameplay.
10. **Firebase Analytics & Logging**: Use the native `logEvent(name, params)` helper in `game.js` to dispatch custom gameplay metrics (e.g. `view_title_screen`, `level_start`, `level_playing`, `game_over`, `level_victory`). Avoid raw calls to native channel wrappers; utilize the helper method in `game.js`.
11. **Crashlytics Error Reporting**: JS-side exceptions are forwarded via `window.onerror` directly to Firebase Crashlytics on the native side. Any custom try-catch blocks in JS should log errors back using `LeaderboardChannel.postMessage(JSON.stringify({ type: 'recordError', message: e.message, stack: e.stack }))` when handling critical runtime operations.
12. **Firebase Performance Monitoring**: High-latency web operations (fetching leaderboard scores and posting score entries) are monitored on the Flutter side under performance traces named `get_leaderboard_scores` and `save_leaderboard_score`. Keep network payloads optimized to maintain fast latency metrics.
